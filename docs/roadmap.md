# Roadmap — опциональные идеи без обязательств

Этот файл — свалка идей для **возможных** вех после M4. Здесь нет сроков и
нет обязательств. Делается только то, что захочется делать. Основной план
M0–M4 живёт в `CLAUDE.md`.

## Северная звезда (north star)

Долгосрочный ориентир — **полноценное управление ригами уровня HiveOS**, не
только графики/метрики: flight sheets, управление кошельками и пулами, настройки
видеокарт (overclock/power limit). HiveOS — **эталон по функциональности и UX**.

Важно: это **аспирация, а не обязательство**. «Эталон» ≠ «обязаны достичь
паритета к дате». Дисциплина M-вех в силе: M0–M4 (в `CLAUDE.md`) не меняются;
управляющие фичи живут здесь как вехи **после** M4, без сроков. Что-то может не
получиться (особенно overclock — см. ниже) — это нормально и заложено.

Хорошая новость: архитектура M0–M4 (абстракции `MinerDriver`/`GpuCollector`,
командный WS-канал, config-state в SQLite — ADR 0002/0003/0005) уже служит
фундаментом для этой цели. Большинство управляющих фич = «богатый config-state +
команды по WS», переделки основы не требуют.

## Принцип

Каждая будущая веха, если за неё возьмёмся, должна остаться самодостаточной:
работающий проверяемый функционал, проект полезен и без следующих вех.

## Идеи

### Поддержка железа

- **AMD GPU-метрики** — вторая реализация границы `GpuCollector` (ADR 0004)
  через `rocm-smi` / sysfs `/sys/class/drm/.../device/hwmon`. Граница уже
  спроектирована, нужна реализация + research (rocm-smi vs sysfs, готовые
  экспортёры). Поднимается, когда AMD-риг реально подключается.
- **Другие майнеры** — новые реализации `MinerDriver` (ADR 0003): T-Rex,
  lolMiner, gminer, TeamRedMiner. Аддитивно, по реальной надобности.
- **System-метрики рига** — CPU, RAM, диск, сеть (node_exporter или свой
  collector).

### Эксплуатация и надёжность

- **Alerts через Alertmanager** — температурные пороги, отвал майнера, потеря
  связи с агентом.
- **Watchdog** — авто-перезапуск зависшего майнера / рига по правилам.
- **Multi-pool failover** — резервные пулы в конфиге майнера.

### Управление (путь к north star, прогрессия после M4)

Естественное продолжение командного канала и config-state. Ложится на текущие
абстракции без переделки основы. Ориентировочный порядок от простого к сложному:

1. **Управление конфигом майнера через UI** — редактировать пул/кошелёк/алго в
   UI (а не править TOML на риге руками), пуш по WS, агент применяет через
   `MinerDriver`. Самый прямой следующий шаг к управлению.
2. **Управление кошельками** — справочник кошельков в config-state (SQLite),
   на которые ссылаются конфиги. Чувствительные данные (пароли пулов) —
   аккуратно, не в метрики, возможно отдельный ADR по хранению секретов.
3. **Flight sheets** — именованные профили (майнер+пул+кошелёк+алго),
   применяемые на риг или группу ригов одним действием (ядро UX HiveOS).
4. **Группы ригов / массовые действия** — применить flight sheet / команду на
   набор ригов сразу.
5. **Overclock / power limit / fan** — ⚠️ самый сложный и рискованный пункт.
   Это **запись в железо** (новая граница `GpuController`, write, в отличие от
   read-only `GpuCollector`), платформо-зависимая (Nvidia NVML/coolbits, AMD
   rocm-smi/sysfs) и с риском для оборудования. **Требует отдельного ADR с
   серьёзным обоснованием** (как уже сказано в `CLAUDE.md`). Не на первых вехах;
   может не быть реализовано вовсе.

### Прочее

- Docker-образ агента как альтернатива systemd-unit (см. ADR 0001 —
  пока отвергнуто).
