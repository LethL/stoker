# ADR 0001: Stoker — приложение поверх ОС, а не дистрибутив

**Дата:** 2026-06-18
**Статус:** Принято

## Контекст

HiveOS, ближайший ориентир по UX, — это OS-образ: собственный загрузочный
дистрибутив с ядром, драйверами, прошивкой. Соблазн «сделать как HiveOS»
тянет в сторону сборки ISO, патчинга ядра, упаковки GPU-драйверов.

## Решение

Stoker — **приложение, устанавливаемое поверх существующей Ubuntu/Debian**.
Не ISO, не образ, не патчи ядра, не сборка драйверов, не initramfs. Агент
ставится как обычный пакет + systemd-unit; сервер — docker-compose.

## Обоснование

- Pet-проект на 3 рига: поддержка OS-образа — несоизмеримая нагрузка
  (сборка, обновления, тестирование железа, секьюрити-патчи ядра).
- Цель — управление **своими** ригами, где ОС уже стоит и настроена.
- Драйверы/прошивка — зона ответственности дистрибутива и вендора, не наша.
- Фокус обучения автора — Python/FastAPI/метрики/React, а не OS-инжиниринг.

## Последствия

- Предполагаем, что на риге уже есть рабочие GPU-драйверы (Nvidia/AMD),
  Python 3.12+, systemd. Документируем требования, не обеспечиваем их сами.
- Не конкурируем с HiveOS по «всё в одном образе» — берём только UX-идеи.
- Установка агента = инструкция + unit-файл, а не «прошить флешку».

## Альтернативы и почему отвергнуты

- **Свой дистрибутив (как HiveOS):** неподъёмно для pet-проекта, вне целей.
- **Docker-образ агента на риге:** усложняет доступ к GPU/pynvml/процессам
  майнера (нужны `--gpus`, проброс устройств); systemd-unit проще и прозрачнее
  для Linux-грамотного пользователя. Не исключено позже, но не сейчас.
