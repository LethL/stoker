# ADR 0004: Абстракция сбора GPU-метрик (Nvidia-first, AMD отложен)

**Дата:** 2026-06-18
**Статус:** Принято

## Контекст

CLAUDE.md (M2a) изначально завязывал сбор GPU-метрик на `pynvml` — а pynvml
работает **только с Nvidia**. Однако реальный парк автора **смешанный
(AMD + Nvidia)**. Для AMD на Linux нужен совсем другой путь сбора
(`rocm-smi` / чтение `/sys/class/drm/.../device/hwmon` sysfs).

Research 0003 дополнительно показал: power и clocks из API майнера (WildRig)
ненадёжны/отсутствуют — значит температуру/fan/power/clock берём из слоя
железа, а не из майнера. Это делает слой сбора GPU-метрик самостоятельным и
vendor-зависимым.

## Решение

Ввести **границу `GpuCollector`** (Python `Protocol`) в `agent/src/stoker_agent/
collectors/`, симметричную [[0003-miner-driver-abstraction]]. Коллектор отдаёт
нормализованную модель `GpuStats` (temp, fan, power, clock, util, mem per GPU),
не привязанную к вендору.

**Реализуем сейчас только `NvidiaCollector` (pynvml)** — первый риг на Nvidia.
**`AmdCollector` (rocm-smi/sysfs) — зафиксированный отложенный пункт:** интерфейс
под него спроектирован, реализация пишется, когда дойдём до AMD-рига, **без
переделки** остального кода.

Агент **авто-определяет** доступные GPU на хосте и поднимает нужный коллектор(ы)
— один бинарь работает на all-Nvidia, all-AMD и смешанном риге.

## Обоснование

- Парк смешанный → привязка к pynvml неполна для реальности автора.
- Порядок развёртывания автора: Nvidia-риг первым, AMD — в последнюю очередь.
  Значит реализуем Nvidia сейчас, AMD фиксируем как вторую реализацию готовой
  границы — это и защищает от привязки, и не раздувает M2a.
- Симметрия с `MinerDriver`: те же принципы (Protocol, нормализованная модель,
  defensive parsing) — меньше когнитивной нагрузки.
- hashrate приходит из `MinerDriver` (vendor-независим), temp/fan/power/clock —
  из `GpuCollector` (vendor-зависим). Граница проходит чисто.

## Последствия

- M2a реализует Nvidia-путь; AMD-путь — будущая веха (поднята из «опционально»
  в «вторая реализация существующей границы, отложена»).
- Единый `/metrics` агента сводит `GpuStats` + `MinerStats` под общими лейблами
  (`rig`, `gpu`, `miner`).
- AMD-specific research (rocm-smi vs sysfs, готовые экспортёры) осознанно
  отложен до момента, когда AMD-риг реально подключается.
- Имена метрик — в стиле dcgm-exporter (см. [[0002-prometheus-for-metrics]]).

## Альтернативы и почему отвергнуты

- **Только pynvml, AMD игнорировать:** не покрывает реальный парк автора.
- **Сразу обе реализации (Nvidia+AMD):** AMD-риг ставится последним — писать и
  поддерживать AMD-сбор до того, как он нужен, = преждевременная работа.
- **Переиспользовать внешний экспортёр (nvidia_gpu_exporter):** валидная
  альтернатива, но агент всё равно работает на риге и уже опрашивает майнер —
  один процесс/scrape-таргет чище, плюс pynvml — заявленная цель обучения.
