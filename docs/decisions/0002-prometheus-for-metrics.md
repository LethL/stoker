# ADR 0002: Prometheus + Grafana для метрик, SQLite для state

**Дата:** 2026-06-18
**Статус:** Принято

## Контекст

Нужно хранить и показывать числовые метрики ригов во времени (hashrate,
температуры, fan, power) и при этом хранить state (конфиги ригов, команды,
события, статус подключения). Это два принципиально разных типа данных:
плотные time-series против реляционного state.

Research 0002 показал: мёртвые аналоги (Minera, cryptoGlance, MiningMonitor)
строили собственное хранилище метрик + кастомные графики (LiteDB/Mongo) — и
сгнили на их поддержке.

## Решение

**Числовые метрики во времени → Prometheus** (scrape `/metrics` агента),
**графики/дашборды/алерты → Grafana**. Всё остальное (state, конфиги,
события, команды) → **SQLite через сервер** (SQLModel).

Граница жёсткая: сервер НЕ хранит time-series, Prometheus НЕ хранит state.

## Обоснование

- Индустриальный стандарт, меньше своего кода, переносимый навык (цель обучения).
- Pull-модель (Prometheus скрейпит агента) — стандарт fleet-мониторинга,
  не требует входящих соединений к серверу от агента ради метрик.
- Grafana снимает с нас весь UI исторических графиков (через iframe в M3).
- SQLite на 3 рига достаточно; миграция на Postgres через SQLModel почти
  бесплатна, если упрёмся.

## Последствия

- Дашборды провизионятся автоматически (Grafana provisioning), JSON вендорим
  в репозиторий — не зависим от remote dashboard ID.
- Имена метрик согласуем со стилем dcgm-exporter для переносимости дашбордов.
- Два хранилища = две точки бэкапа/эксплуатации; принято осознанно.
- Метрики майнера (hashrate, shares) и метрики железа (temp/fan/power) сводятся
  в Prometheus по общим лейблам (`rig`, `gpu`).

## Альтернативы и почему отвергнуты

- **Своё хранилище time-series в SQLite + свои графики:** ровно то, на чём
  умерли аналоги; огромный объём кода, плохо масштабируется, не учит стандарту.
- **TimescaleDB/InfluxDB:** тяжелее Prometheus для нашего масштаба, без Grafana
  всё равно пришлось бы; Prometheus+Grafana — связка по умолчанию в отрасли.
- **Grafana Agent как коллектор:** EOL (ноя 2025); используем plain Prometheus
  scrape (или Alloy, если коллектор когда-то понадобится).
